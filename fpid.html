Technical — How it links

Visitor = request. The "visitor" in the animation is any system (e.g. P13N or an external app) that sends a request carrying identifiers (email, user_id, unique_client_id). That request is the "knock."

The door = REST or Thrift. REST API (p13n) is used by personalization; REST API (external) is used by other clients. Both paths lead into People Service; internally, Thrift API is the common layer that does the work.

Thrift does the work. It resolves identity (match or create), updates the primary store (Aerospike), and publishes an identity_event (e.g. NEW_IDENTITY or UPDATE_IDENTITY) to Kafka. Consumers (people-consumer-elasticsearch, people-consumer-snowflake, people-consumer-zync, etc.) subscribe to identity_events and keep their systems in sync.

End-to-end: Request → REST/Thrift door → People Service (Thrift) → Aerospike + identity_events → consumers update ES, Snowflake, Zync, etc.

Technical — People-consumers

People-consumers read identity_events and update downstream: DC user traits (known_to_zeta, reachable_channels), GeoCode, Elasticsearch, Snowflake, Zync. Each consumer enriches then writes back via People APIs / bulk_store_person.
